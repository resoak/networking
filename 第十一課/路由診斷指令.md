# 路由診斷指令

## 這一頁在記什麼

錄音 22 補充了幾個常用網路診斷指令。這些指令可以用來檢查主機是否可達、路徑經過哪些節點、本機 ARP cache 與 routing table 狀態。

## ping

`ping` 用來測試目的主機是否有回應。

常見用途：

- 確認對方主機是否可達。
- 觀察延遲時間。
- 初步判斷網路是否中斷。

注意：有些設備或防火牆會擋 ICMP，所以 `ping` 不通不一定代表對方真的不存在。

## tracert

`tracert` 用來追蹤封包到目的地中間經過哪些 Router。

它的基本想法是：

1. 先送出 TTL=1 的封包。
2. 第一台 Router 讓 TTL 歸零後回覆。
3. 再送 TTL=2、TTL=3。
4. 逐步顯示中間節點，直到目的地或達到限制。

所以 `tracert` 很像把路徑一站一站列出來。

## arp

`arp -a` 可以查看本機 ARP cache。

它會列出：

- IP address。
- 對應的 MAC address。
- 動態或靜態項目。

ARP cache 可以幫助我們確認同一區域網路中，IP 與實體位址的對應是否合理。

## route

`route print` 可以查看本機 routing table。

它可以用來觀察：

- 目的網路。
- subnet mask。
- gateway。
- interface。
- metric。

必要時也可以用 `route add` 或 `route delete` 新增或刪除本機路由，但課堂重點是先看懂表格，不是背指令語法。

## pathping

`pathping` 可以想成加強版 `tracert`。

它不只列出路徑，還會花較多時間統計路徑上各節點的封包遺失比例。當你懷疑某段路徑不穩、丟包嚴重時，`pathping` 會比單純 `tracert` 提供更多線索。

## 指令整理

| 指令 | 主要用途 |
|---|---|---|
| `ping` | 測試目的主機是否有回應 |
| `tracert` | 追蹤封包經過哪些 Router |
| `arp -a` | 查看 IP 與 MAC address 的對應 |
| `route print` | 查看本機 routing table |
| `pathping` | 追蹤路徑並統計丟包情況 |

## 實務應用

錄音 22 提到了實際的應用場景：

> `ping`：有沒有通。
> `tracert`：走哪條路。
> `arp`：IP 對到哪個 MAC。
> `route`：本機查表怎麼走。
> `pathping`：哪一段可能在丟包。

**網路診斷策略：**

1. **連通性測試**：首先測試基本的連通性
2. **路徑追蹤**：確認封包走過的完整路徑
3. **ARP 查詢**：確認 IP 地址與 MAC 地址的對應
4. **路由表檢查**：確認本機的路由表設定
5. **封包遺失分析**：統計各節點的封包遺失情況

**診斷流程：**

1. **初步測試**：使用 `ping` 測試基本連通性
2. **詳細追蹤**：使用 `tracert` 追蹤完整路徑
3. **節點檢查**：使用 `arp -a` 檢查各節點的 ARP 設定
4. **路由檢查**：使用 `route print` 檢查路由表
5. **深入分析**：使用 `pathping` 詳細分析封包遺失情況

## 實務建議

**網路診斷的順序：**

1. **先簡單測試**：使用 `ping` 測試基本連通性
2. **再詳細追蹤**：使用 `tracert` 確認路徑
3. **檢查 ARP**：使用 `arp -a` 確認 IP-MAC 對應
4. **檢查路由**：使用 `route print` 確認路由表
5. **深入分析**：使用 `pathping` 分析封包遺失

**為什麼要這樣順序？**

- `ping` 最快，能快速確認基本連通性
- `tracert` 需要較長時間，但可以確認完整路徑
- `arp -a` 快速確認各節點的對應關係
- `route print` 顯示完整的路由設定
- `pathping` 需要較長時間，但提供最詳細分析

- `arp`：IP 對到哪個 MAC。
- `route`：本機查表怎麼走。
