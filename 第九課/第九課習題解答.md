## 一、填充題

1. TCP/IP 傳輸層的協定以 **TCP** 與 **UDP** 兩種傳送模式最為常見。

2. TCP/IP 架構下傳輸層是以所謂的 **埠編號（Port Number）** 對服務類型進行定址工作。

3. TCP 執行 Checksum 值運算時需要同時運用 **虛擬標頭（Pseudo Header）**、**TCP 標頭**、**TCP 資料** 進行運算。

4. TCP 標頭內的 **Window／視窗欄位**，主要是記錄資料封包在傳送或接收後，傳送主機內的緩衝區還剩下多少可以裝載封包的資料空間。

5. 每次 TCP 連線建立之初，都必須將 TCP 標頭內控制旗標的 **SYN** 位元設為 1，用以執行同步溝通；TCP 連線結束則會將 **FIN** 位元設為 1。

6. TCP 標頭選擇項的基本欄位主要有三個，分別是 **Kind／種類**、**Length／長度** 與 **Value 或 Data／內容資料**。

## 二、選擇題

1. 下列何者不是傳輸層的協定？  
   **答案：A，ICMP**  
   ICMP 屬於網路層相關協定，不是傳輸層協定。

2. 關於 TCP 與 UDP 比較的敘述，下列何者有誤？  
   **答案：B，TCP 處理時間較短**  
   TCP 需要連線建立、確認、重傳、流量控制等機制，處理時間通常比 UDP 長。

3. 下列何者以 TCP 為傳輸層協定？  
   **答案：C，POP3**  
   POP3 常用 TCP 連線，預設埠號為 110。

4. 下列何者一般以 UDP 為傳輸層協定？  
   **答案：D，DNS**  
   DNS 一般查詢多使用 UDP 53；區域傳送或大型回應也可能使用 TCP。

5. 下列公認埠編號與服務應用項目的配對何者有誤？  
   **答案：A，FTP: 19**  
   FTP 常見控制連線埠號是 21，資料連線常用 20；19 不是 FTP 的標準埠號。

6. 關於 UDP 的敘述何者有誤？  
   **答案：C，會檢測是否成功傳送**  
   UDP 是 best-effort、非可靠傳輸，不負責確認是否成功送達。

7. 下列何者為虛擬標頭的功能？  
   **答案：B，輔助進行 Checksum 的運算**  
   虛擬標頭不會真正傳送出去，主要用來讓 TCP/UDP Checksum 檢查時也涵蓋來源 IP、目的 IP、協定號與長度等資訊。

8. 下列何者不是 TCP 協定的特性？  
   **答案：此題選項疑似誤植，A、B、C 都是 TCP 特性。**  
   TCP 具備高可靠度、傳送後需確認回覆、傳送前需先建立連線；若原題其實是問「何者是 TCP 特性」，則答案會是 **D，以上皆是**。

9. 下列何者不是回應號碼的功能？  
   **答案：C，傳輸過程發生延遲的記錄**  
   ACK Number 主要表示下一個期望收到的序列號碼，也可用於判斷資料是否成功送達，但不負責記錄延遲。

10. 下列何者不是 TCP 協定中傳送方發現錯誤出現的方式？  
    **答案：C，Window**  
    傳送方可透過計時器逾時、重複 ACK 等方式判斷可能遺失；Window 主要用於流量控制。

11. TCP 協定運用各工作節點內建記憶體作為緩衝區，使節點同一時間可處理多個訊息，該緩衝區稱為？  
    **答案：C，Window**

12. TCP 連線建立之初，首先送出的同步溝通訊號，是設定 TCP 控制旗標中的？  
    **答案：B，SYN**

13. TCP 標頭選擇項中，作為填充位元用的是？  
    **答案：D，Padding**  
    Padding 用來讓 TCP 標頭長度補齊為 32 位元的整數倍。補充：TCP Options 中的 No-operation（NOP）也常用於選項對齊。

14. TCP 使用 SACK 機制時，如有一封包遺失，該封包的序列號碼會記錄於何欄位？  
    **答案：B，ACK**  
    嚴格來說，遺失資料的起點會由累積 ACK Number 表示；SACK 的 Left Edge 與 Right Edge 則記錄「已收到但不連續」的資料區塊範圍，讓傳送端推論中間缺口。

### 3. 請詳細說明埠編號的功能、分類與定義方式，並說明 UDP 與 TCP 的保留埠編號

**埠編號（Port Number）** 是傳輸層用來辨識主機內部不同應用程式或服務的編號。IP 位址用來找到哪一台主機，Port Number 則用來找到該主機上的哪一個服務或程式。傳輸層常用「來源 IP、來源 Port、目的 IP、目的 Port、傳輸協定」來識別一條通訊連線或資料流。

Port Number 長度為 **16 bits**，範圍是 **0～65535**，常見分類如下：

| 分類 | 範圍 | 說明 |
|---|---:|---|
| Well-known Ports／公認埠 | 0～1023 | 分配給常見標準服務，例如 HTTP 80、HTTPS 443、DNS 53、SMTP 25 |
| Registered Ports／登錄埠或使用者埠 | 1024～49151 | 分配給特定應用程式或廠商服務 |
| Dynamic/Private/Ephemeral Ports／動態或私有埠 | 49152～65535 | 用戶端暫時連線常使用的臨時埠 |

TCP 與 UDP 都使用 16-bit Port Number，但 **TCP 與 UDP 的埠號空間是分開管理的**。因此同一個數字可以同時存在於 TCP 與 UDP，例如 DNS 常見使用 UDP 53，也可在特定情況下使用 TCP 53。保留埠編號通常由 IANA 管理，用來避免常見服務之間發生衝突。

---

### 4. 請針對 UDP 標頭內各欄位所佔位元數及功能作詳細說明

UDP Header 固定長度為 **8 bytes = 64 bits**。

| 欄位 | 長度 | 功能 |
|---|---:|---|
| Source Port | 16 bits | 來源應用程式的埠號；若不需要回覆，可為 0 |
| Destination Port | 16 bits | 目的應用程式或服務的埠號 |
| Length | 16 bits | UDP Header 加上 UDP Data 的總長度，單位為 byte |
| Checksum | 16 bits | 用於檢查 UDP 虛擬標頭、UDP 標頭與資料是否發生錯誤 |

UDP Header 很小，因此處理速度快，但也因為缺少序號、ACK、重傳與流量控制，所以可靠性較低。

### 5. 何謂虛擬標頭？請說明功能

**虛擬標頭（Pseudo Header）** 是 TCP 或 UDP 計算 Checksum 時暫時加入的資料結構，**不會真的封裝進 TCP/UDP 封包中傳送**。

IPv4 中常見的虛擬標頭包含：

| 欄位 | 功能 |
|---|---|
| Source IP Address | 確認來源主機 |
| Destination IP Address | 確認目的主機 |
| Zero | 補 0 對齊欄位 |
| Protocol | 標示上層協定，例如 TCP = 6、UDP = 17 |
| TCP/UDP Length | 標示 TCP 或 UDP 資料長度 |

它的主要功能是讓 Checksum 不只檢查傳輸層標頭與資料，也能檢查 IP 位址與協定號是否正確，避免封包被送到錯誤的主機或錯誤的傳輸層協定時卻未被發現。

---

### 6. 請說明 UDP 的 Checksum 值運算機制

UDP Checksum 的運算流程如下：

1. 先建立 UDP 虛擬標頭，包含來源 IP、目的 IP、協定號與 UDP 長度。
2. 將 UDP Header 中的 Checksum 欄位先設為 0。
3. 將 **虛擬標頭 + UDP Header + UDP Data** 接在一起。
4. 若總長度不是 16 bits 的整數倍，最後補 0。
5. 以 16-bit 為單位進行一補數加法（one's complement sum）。
6. 若最高位元產生進位，需回捲加到最低位元。
7. 將加總結果再取一補數，即為 Checksum。
8. 接收端用同樣方式重新計算；若結果正確，整體加總應為 `0xFFFF`。

補充：IPv4 中 UDP Checksum 可選擇不用；IPv6 中 UDP Checksum 是必要的。

### 8. 針對 TCP 架構的封包傳送，請說明序列號碼與回應號碼間的互動關係

TCP 是以 **byte stream** 為基礎運作，因此 Sequence Number 不是單純的「第幾個封包」，而是資料流中「某個 byte 的位置」。

- **Sequence Number**：表示此 TCP Segment 所攜帶資料的第一個 byte 編號。
- **Acknowledgment Number**：表示接收端下一個期望收到的 byte 編號。
- 若一方送出 `Seq = X`、資料長度為 `N bytes`，接收端成功收到後，通常會回覆 `ACK = X + N`。
- 若是 SYN 或 FIN，雖然不一定攜帶一般資料，但在 TCP 序號空間中會消耗 1 個序列號碼，所以回覆時 ACK 會加 1。
- TCP 的 ACK 通常是累積確認，代表「此 ACK 之前的資料都已收到」。

### 9. 針對圖 9.10 所示的兩種情況，說明情況發生情形及處理機制

圖 9.10 說明 TCP 傳送時最基本的兩種錯誤情況：**資料封包本身遺失**與**回應訊息（ACK）遺失**。兩者對傳送端來說都會表現為「在計時器時間內沒有收到對方 ACK」，因此都會觸發重傳。

| 情況 | 發生情形 | TCP 處理方式 |
|---|---|---|
| 圖 9.10(a)：封包遺失 | 傳送端送出資料封包並啟動計時器，但封包在傳送途中遺失，接收端沒有收到資料，因此也不會送出對應 ACK | 傳送端等到計時器逾時仍未收到 ACK，就重新傳送同一個封包；接收端收到重傳封包後再回 ACK，傳送端收到 ACK 後取消計時器 |
| 圖 9.10(b)：回應訊息遺失 | 接收端已經正確收到資料封包，也送出 ACK，但 ACK 在回傳途中遺失，導致傳送端以為資料可能沒有成功送達 | 傳送端逾時後會重傳原資料封包；接收端若再次收到同一資料，會依 Sequence Number 判斷為重複封包，不會重複交給上層處理，只需重新送出 ACK |

重點是：TCP 不一定能立刻分辨「資料沒送到」或「ACK 沒回來」，所以先用 **計時器逾時（timeout）** 來判定需要重傳，再靠 **Sequence Number** 避免接收端把重複資料當成新資料處理。

### 10. 回應號碼除了用以判斷封包傳送成功與否之外，還有什麼含意？

ACK Number 還代表 **接收端下一個期望收到的資料 byte 序號**。例如接收端回覆 `ACK = 5001`，表示序號 5000 以前的資料都已收到，下一個希望收到的資料從 5001 開始。

因此 ACK Number 也具備下列意義：

- 表示資料接收進度。
- 告訴傳送端下一段資料應從哪個 Sequence Number 開始。
- 若多次收到相同 ACK，可能代表中間某段資料遺失。
- 搭配滑動視窗時，可讓傳送端移動可傳送範圍。

### 11. 請比較圖 9.11、圖 9.12 與圖 9.13 的運作機制及其優缺點

圖 9.11～圖 9.13 都是在說明 TCP 為了提高傳輸效率，如何從「每送一個封包就等一次 ACK」逐步改進成「利用緩衝區連續傳送」與「合併回應」。

| 機制 | 運作方式 | 優點 | 缺點 |
|---|---|---|---|
| 圖 9.11：一般通訊傳輸情形 | 傳送端送出封包 1 後，必須等接收端回覆「封包 1 回覆」才送封包 2；每個封包都依序等待 ACK 後才繼續 | 流程最簡單，容易確認每一個封包是否成功送達，控制與實作成本低 | 等待時間很長，傳送端常處於閒置狀態，頻寬利用率差，整體傳輸效率低 |
| 圖 9.12：運用緩衝區進行訊息傳送 | 兩端節點利用內建記憶體作為緩衝區，傳送端不必等前一個 ACK 完全回來，就能在可用範圍內連續送出多個封包；接收端也可暫存尚未處理的資料 | 可讓多個封包與多個 ACK 在同一段時間內交錯進行，大幅降低等待時間，提高傳輸效率 | 需要額外緩衝記憶體與序號管理；若緩衝區不足或封包遺失，仍需靠 ACK、逾時與重傳處理 |
| 圖 9.13：合併回應／累積 ACK | 接收端收到多個連續封包後，不必每個封包都立刻各回一個 ACK，而是用一次 ACK 表示前面連續資料都已收到；例如右圖中 `Seq: 20000`、`21500`、`23000`、`24500` 各傳 `1500 Bytes`，最後回 `ACK: 26000`，代表 26000 以前的資料都已正確收到 | 減少 ACK 封包數量，降低網路負擔與來回傳送時間，整體效率最高 | 如果中間某個封包遺失，累積 ACK 只能指出「下一個期望收到的位置」，無法單靠一般 ACK 詳細指出後面哪些封包已收到；因此仍需搭配重傳機制或 SACK 等進一步機制 |

總結來說，圖 9.11 著重可靠但效率低；圖 9.12 利用緩衝區與類似滑動視窗的概念提升同時處理量；圖 9.13 再透過累積式 ACK 合併回應，減少 ACK 數量並提升整體傳輸效率。

### 12. 何謂滑動視窗？其運用的單位可能有哪幾種？

**滑動視窗（Sliding Window）** 是 TCP 用來進行流量控制與提升傳輸效率的機制。傳送端不必每送一個封包就等待 ACK，而是可以在視窗允許範圍內連續傳送多筆資料。當接收端回覆 ACK 後，視窗範圍會往前移動，因此稱為「滑動」。

滑動視窗的單位可能有：

1. **Byte／位元組**：TCP 的 Window Size 欄位實際上以 byte 為單位。
2. **Segment／資料片段或封包數**：教學或簡化模型中常用封包數表示。
3. **Frame／訊框數**：資料鏈結層的滑動視窗協定常以 frame 為單位。

### 13. 請觀察圖 9.17～圖 9.22，針對 TCP 標頭各欄位值的互動進行分析與說明

圖 9.17～圖 9.22 是以 FTP 控制連線為例，觀察 206 節點（`140.134.28.206`）與 110 節點（`140.134.36.110`）之間 TCP 封包的 **Sequence Number、Acknowledgment Number、Window** 如何隨資料傳送而改變。

#### 主要互動規則

1. **Sequence Number 代表本端送出資料的起始 byte 位置**  
   若某端送出 `N bytes` 資料，下一次同方向再送資料時，Sequence Number 會增加 `N`。

2. **Acknowledgment Number 代表下一個期望收到的 byte 位置**  
   接收端收到對方 `Seq = X`、資料長度 `N bytes` 的封包後，會回覆：

   ```text
   ACK = X + N
   ```

3. **TCP 是雙向傳輸，兩個方向的 Sequence Number 要分開計算**  
   206 → 110 有一組序列號；110 → 206 也有另一組序列號。ACK Number 是用來確認「對方」送來的資料，不是確認自己送出的資料。

4. **Window 表示接收端剩餘緩衝空間**  
   題目說明中假設接收後暫不清空緩衝區，因此收到資料後 Window 會依收到的資料長度減少。

#### 依圖 9.17～圖 9.22 逐封包分析

| 圖號 | 傳送方向 | 主要內容 | Seq | 資料長度 | ACK | Window 變化／意義 |
|---|---|---|---:|---:|---:|---|
| 圖 9.17 | 206 → 110 | FTP 伺服器訊息，例如 `220 Serv-U...` | `10072715` | `49 bytes` | `22385505` | 206 宣告自己的接收視窗約為 `8760` |
| 圖 9.18 | 110 → 206 | 純 ACK，確認圖 9.17 | `22385505` | `0 bytes` | `10072764` | `10072764 = 10072715 + 49`；110 收到 49 bytes 後視窗減少 |
| 圖 9.19 | 206 → 110 | 206 繼續送 FTP 資料 | `10072764` | `60 bytes` | `22385505` | Seq 接續圖 9.18 的 ACK，表示從 `10072764` 繼續送 |
| 圖 9.20 | 110 → 206 | `USER vision`，並確認圖 9.19 | `22385505` | `13 bytes` | `10072824` | `10072824 = 10072764 + 60`；Window 約 `8651 = 8711 - 60` |
| 圖 9.21 | 206 → 110 | `OK, send password...`，並確認圖 9.20 | `10072824` | `37 bytes` | `22385518` | `22385518 = 22385505 + 13`；Window 約 `8747 = 8760 - 13` |
| 圖 9.22 | 110 → 206 | `PASS Pixel`，並確認圖 9.21 | `22385518` | `12 bytes` | `10072861` | `10072861 = 10072824 + 37`；Window 約 `8614 = 8651 - 37` |

#### 重點說明

- 圖 9.18 的 `ACK = 10072764`，表示 110 已收到 206 從 `10072715` 開始的 `49 bytes`，所以下一次期待 206 從 `10072764` 開始傳送。
- 圖 9.19 中 206 的 `Seq = 10072764`，正好接續圖 9.18 的 ACK，表示 TCP 會依對方確認號繼續傳資料。
- 圖 9.20 同時做兩件事：一方面用 `ACK = 10072824` 確認 206 的 `60 bytes`，另一方面 110 自己也從 `Seq = 22385505` 送出 `13 bytes`。
- 圖 9.21 再用 `ACK = 22385518` 確認 110 的 `13 bytes`，並由 206 從 `Seq = 10072824` 送出 `37 bytes`。
- 圖 9.22 用 `ACK = 10072861` 確認 206 的 `37 bytes`，同時 110 從 `Seq = 22385518` 送出密碼資料 `PASS Pixel`，長度約 `12 bytes`。

因此，圖 9.17～圖 9.22 的核心觀念是：**TCP 的 ACK Number 永遠表示「下一個期望收到的 byte 序號」；Sequence Number 則表示「本封包資料從哪個 byte 序號開始」。雙方可以在同一個 TCP 封包中同時傳送資料與確認對方資料，所以 Seq 與 ACK 會在兩個方向上交錯變化。**

### 15. 針對 HLEN 欄位，為何有時又稱為 Data Offset？兩者是否代表相同意義？

HLEN 是 **Header Length**，表示 TCP Header 的長度；Data Offset 則表示「從 TCP Segment 起點到資料區起點的位移」。

因為 TCP Header 長度可能因 Options 欄位而改變，而資料區正好位於 TCP Header 之後，所以：

```text
Data Offset = TCP Header Length
```

兩者本質上代表相同意義，都是用來告訴接收端 TCP Header 到哪裡結束、Data 從哪裡開始。此欄位單位是 **32-bit word**；若值為 5，代表 Header 長度為 `5 × 4 = 20 bytes`。

### 16. 請說明 TCP 標頭 8 個控制旗標 C、E、U、A、P、R、S、F 的實際運用場合

| 旗標 | 全名 | 用途 |
|---|---|---|
| C | CWR, Congestion Window Reduced | 表示傳送端已收到壅塞通知並降低壅塞視窗 |
| E | ECE, ECN-Echo | 用於 ECN 壅塞通知；表示網路發生壅塞但未必丟包 |
| U | URG, Urgent | Urgent Pointer 有效，表示封包含緊急資料 |
| A | ACK, Acknowledgment | ACK Number 有效；連線建立後大多數 TCP 封包都會設為 1 |
| P | PSH, Push | 要求接收端盡快將資料交給應用程式，不必等待緩衝區累積更多資料 |
| R | RST, Reset | 強制重設或中斷連線，常見於連線錯誤或拒絕連線 |
| S | SYN, Synchronize | 建立連線時同步雙方初始序列號碼 |
| F | FIN, Finish | 表示發送方已無資料要傳，用於正常結束連線 |

### 17. 請比較 UDP 虛擬標頭與 TCP 虛擬標頭的差異

| 比較項目 | TCP 虛擬標頭 | UDP 虛擬標頭 |
|---|---|---|
| 是否實際傳送 | 不會實際傳送 | 不會實際傳送 |
| 主要用途 | 輔助 TCP Checksum | 輔助 UDP Checksum |
| 共同欄位 | 來源 IP、目的 IP、Zero、Protocol、Length | 來源 IP、目的 IP、Zero、Protocol、Length |
| Protocol 值 | 6 | 17 |
| Length 意義 | TCP Header + TCP Data 長度 | UDP Header + UDP Data 長度 |
| Checksum 必要性 | TCP Checksum 必要 | IPv4 中 UDP Checksum 可選；IPv6 中必要 |

兩者格式概念幾乎相同，差別主要在 Protocol 值、Length 所代表的資料，以及後續實際計算時搭配的是 TCP Header 或 UDP Header。

### 18. 針對 TCP 連線的啟動與結束，說明封包欄位的互動

TCP 連線的啟動與結束，主要可從 **控制旗標（SYN、ACK、FIN）**、**Sequence Number** 與 **Acknowledgment Number** 的互動來觀察。ACK Number 的意義不是「我收到的序號」，而是 **下一個期望收到的序號**。

#### TCP 連線啟動：三向交握

圖 9.27～圖 9.29 以 110 節點與 206 節點建立 FTP 控制連線為例：

| 步驟 | 方向 | 控制旗標 | Seq | Ack | 欄位互動說明 |
|---|---|---|---:|---:|---|
| 1 | 110 → 206 | `SYN = 1`、`ACK = 0` | `22385504` | `0` | 110 節點提出連線要求，並送出自己的初始序列號碼 |
| 2 | 206 → 110 | `SYN = 1`、`ACK = 1` | `10072714` | `22385505` | 206 節點確認收到 110 的 SYN；因 SYN 會佔用 1 個序號，所以 `Ack = 22385504 + 1` |
| 3 | 110 → 206 | `ACK = 1` | `22385505` | `10072715` | 110 節點確認收到 206 的 SYN；因 206 的 SYN 也佔用 1 個序號，所以 `Ack = 10072714 + 1` |

因此三向交握可以簡化為：

```text
110 → 206：SYN
206 → 110：SYN + ACK
110 → 206：ACK
```

完成第三步後，雙方已同步彼此的初始序號，TCP 連線正式建立，後續即可開始傳送 FTP 資料。

#### TCP 連線結束：四次揮手

圖 9.31～圖 9.34 以 20 節點與 77 節點結束 FTP 連線為例：

| 步驟 | 方向 | 控制旗標 | Seq | Ack | 欄位互動說明 |
|---|---|---|---:|---:|---|
| 1 | 20 → 77 | `FIN = 1`、`ACK = 1` | `1335137020` | `8501483` | 20 節點表示自己資料已送完，提出終止連線要求 |
| 2 | 77 → 20 | `ACK = 1` | `8501483` | `1335137021` | 77 節點確認收到 20 的 FIN；因 FIN 會佔用 1 個序號，所以 `Ack = 1335137020 + 1` |
| 3 | 77 → 20 | `FIN = 1`、`ACK = 1` | `8501483` | `1335137021` | 77 節點也表示自己資料已送完，送出 FIN；因第 2 步只是純 ACK，未帶資料，所以 Seq 仍為 `8501483` |
| 4 | 20 → 77 | `ACK = 1` | `1335137021` | `8501484` | 20 節點確認收到 77 的 FIN；因 77 的 FIN 也佔用 1 個序號，所以 `Ack = 8501483 + 1` |

因此四次揮手可以簡化為：

```text
20 → 77：FIN + ACK
77 → 20：ACK
77 → 20：FIN + ACK
20 → 77：ACK
```

TCP 結束連線通常需要四次，是因為 TCP 是 **雙向資料流**。一方送出 FIN 只代表「我這個方向沒有資料要送了」，不代表對方也已經送完；所以對方要先 ACK，再等自己的資料也送完後送出 FIN，最後再由原本要求結束的一方回 ACK。

#### 核心規則

```text
收到 SYN 後，回 ACK = 對方 SYN 的 Seq + 1
收到 FIN 後，回 ACK = 對方 FIN 的 Seq + 1
```

也就是說，`SYN` 與 `FIN` 雖然不一定攜帶一般資料，但都會在 TCP 序號空間中消耗 1 個 sequence number。

### 19. 圖 9.30 中，第二個封包與第三個封包之間，接收端是否還可以對發送端傳送資料？要求端是否還會傳送回應號碼？

可以。圖 9.30 描述的是 TCP 結束連線時的 **半關閉（half-close）** 狀態：要求端先表示「我不再送資料了」，但接收端到要求端這個方向尚未立刻關閉。

依圖中的序號與回應號碼可整理如下：

| 步驟 | 方向 | 封包內容 | 意義 |
|---|---|---|---|
| 1 | 要求端 → 接收端 | `Seq: X`、`Ack: Y`、`A, F` | 要求端送出 `FIN`，表示自己已無資料要傳；`FIN` 會消耗 1 個序號 |
| 2 | 接收端 → 要求端 | `Seq: Y`、`Ack: X+1`、`A` | 接收端先確認收到要求端的 `FIN`，所以回 `Ack = X + 1` |
| 3 | 接收端 → 要求端 | `Seq: Y`、`Ack: X+1`、`A, F` | 接收端確認自己的資料也都處理完後，才送出自己的 `FIN` |
| 4 | 要求端 → 接收端 | `Seq: X+1`、`Ack: Y+1`、`A` | 要求端確認收到接收端的 `FIN`，所以回 `Ack = Y + 1` |

因此，在 **第二個封包與第三個封包之間**，接收端仍然可以對要求端傳送尚未送完的資料；要求端雖然已經送出 `FIN`，不再傳送一般資料，但仍必須繼續接收資料，並針對接收端送來的資料回覆 ACK。等接收端也確定資料都送完後，才會送出第三個 `FIN + ACK`，最後由要求端回第四個 ACK 結束連線。

### 20. 每個 TCP 連線建立之初，如何設定最大傳送容量？可否擷取封包分析？

TCP 連線建立初期會透過 **MSS（Maximum Segment Size）選項** 設定單一 TCP Segment 可承載的最大資料量。MSS 會放在 TCP 三向交握的 SYN 封包中。

常見設定方式：

```text
TCP SYN 封包
Options:
  Maximum Segment Size: 1460 bytes
```

在乙太網路常見 MTU 為 1500 bytes，若使用 IPv4 且無額外 options：

```text
MSS = MTU - IP Header - TCP Header
MSS = 1500 - 20 - 20 = 1460 bytes
```

實際分析方式可以使用 Wireshark：

1. 開啟封包擷取。
2. 使用瀏覽器連到任一網站。
3. 篩選：`tcp.flags.syn == 1`
4. 點選 SYN 或 SYN+ACK 封包。
5. 展開 TCP Header 的 Options。
6. 觀察 `Maximum Segment Size`、`SACK Permitted`、`Window Scale` 等欄位。

MSS 是雙方各自宣告「自己能接收的最大 TCP payload」，實際傳輸時會依對方宣告值與路徑 MTU 等條件決定。

### 21. 何謂 SACK？有何功能？處理機制如何運作？為何需要這項功能？

**SACK（Selective Acknowledgment，選擇性確認）** 是 TCP 的一種選項，用來改善傳統累積 ACK 在多個封包遺失時效率不足的問題。

#### 功能

傳統 TCP ACK 只能告訴傳送端「從開頭到哪裡為止都已經連續收到」。如果中間某一段遺失，但後面的資料其實已經收到，傳統 ACK 無法精確告訴傳送端後面哪些區塊已成功到達。

SACK 可以讓接收端明確指出：

```text
哪些不連續的資料區塊已經收到
```

傳送端就能只重傳缺少的部分，而不用從遺失處之後全部重傳。

#### 處理機制

1. 連線建立時，雙方在 SYN 封包中使用 `SACK Permitted` 選項表示支援 SACK。
2. 當接收端發現資料不連續，例如中間某段遺失但後面資料已到達，會送出累積 ACK，指出第一個缺少的位置。
3. 同時在 SACK Option 中加入一組或多組區塊範圍：
   - **Left Edge**：已收到區塊的起始序列號碼。
   - **Right Edge**：已收到區塊結束後的下一個序列號碼。
4. 傳送端根據 ACK 與 SACK blocks 判斷哪些資料已收到、哪些資料缺漏。
5. 傳送端優先重傳缺漏的 Segment，避免不必要的重傳。

#### 為何需要 SACK

SACK 對於下列情況特別有用：

- 高延遲網路。
- 頻寬很大但少量丟包的網路。
- 同一個視窗內有多個 Segment 遺失。
- 需要減少不必要重傳，提高有效吞吐量。

簡單說，SACK 的目的就是讓 TCP 在發生遺失時可以「只補缺的」，而不是「從缺的地方後面全部重送」。
