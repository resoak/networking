## 導覽

- 傳輸層在 OSI 七層中位於 **網路層** 之上，負責讓主機上的應用程式能正確收送資料。
- 這一課的核心是：
  - `TCP`：重視可靠性、順序、確認、重傳。
  - `UDP`：重視速度與即時性，不保證送達。
  - `Port number`：讓同一台主機上的不同程式能被區分。
  - `Sequence number / ACK number`：TCP 用來確認資料順序與是否需要重傳。

## 課堂主軸

老師一開始先接續前面的 IP 位址觀念：`IP address` 可以幫資料找到「哪一台主機」，但還不夠解決所有問題。

還需要傳輸層處理：

- 這份資料要交給主機上的哪一個應用程式？
- 傳送過程中資料有沒有遺失？
- 資料順序有沒有錯？
- 如果對方沒有收到，要不要重傳？
- 這個應用比較在意可靠性，還是比較在意即時性？

所以傳輸層提供兩個很重要的協定：`TCP` 與 `UDP`。

## TCP 與 UDP 的基本差異

| 項目 | TCP | UDP |
|---|---|---|
| 全名 | Transmission Control Protocol | User Datagram Protocol |
| 連線方式 | Connection-oriented，有連線概念 | Connectionless，無連線概念 |
| 是否保證送達 | 會確認、會重傳 | 不保證送達 |
| 是否保證順序 | 會用序號維持順序 | 不處理順序保證 |
| 控制資訊 | 較多，Header 較複雜 | 較少，Header 簡單 |
| 速度 | 較慢但可靠 | 較快但不可靠 |
| 適合情境 | 網頁、檔案下載、Email、遠端登入 | 串流、語音、影片、即時資料、部分控制訊號 |

### 怎麼選

- 如果資料**不能缺、不能錯、不能亂序**：通常選 `TCP`。
- 如果資料比較在意**即時送到**，少量遺失可接受：通常選 `UDP`。
- 老師用影片、語音、YouTube 這類資料說明：
  - 使用者通常比較在意「順不順、會不會卡」。
  - 偶爾少一點資料，可能只是畫面或聲音短暫不完整。
  - 如果每一小段都要求確認與重傳，反而會造成延遲。

## TCP：可靠但成本較高

`TCP` 的重點是「可靠傳輸」。

它會做幾件事：

- 傳送前先建立連線。
- 傳送資料時確認對方是否收到。
- 如果資料遺失或錯誤，要求重新傳送。
- 用序號維持資料順序。
- 透過緩衝區與視窗機制控制傳送速度。

### 三向交握

老師用「握手」比喻 TCP 建立連線：正式傳資料前，雙方要先確認彼此都在、也準備好。

三向交握可以簡化成：

1. 傳送端送出 `SYN`：我想建立連線。
2. 接收端回 `SYN + ACK`：我收到，也同意建立連線。
3. 傳送端再回 `ACK`：我知道你準備好了。

完成後，資料才開始傳送。

### 為什麼 TCP 比較慢

TCP 需要額外處理：

- 連線建立。
- ACK 確認。
- 錯誤檢查。
- 遺失時重傳。
- 順序整理。
- 流量控制與緩衝區管理。

這些機制提高可靠性，但也增加等待時間與處理成本。

## UDP：快速但不保證可靠

`UDP` 的特色是「盡力送出」。

它不先建立連線，也不保證對方一定收到。

老師反覆強調：

- UDP 的做法比較像 **fire and forget**。
- 傳送端把資料送出去後，不太管後面發生什麼。
- 如果接收端沒收到，UDP 本身不負責重傳。
- 因為少了許多確認流程，所以速度比較快、延遲比較低。

### UDP 適合的情境

- 即時影音串流。
- 語音通話。
- 遊戲或即時互動。
- 某些網路狀態或控制訊號。
- DNS 查詢這類「問一次沒回，再問一次也可以」的情境。

老師提到：現在網路上的影音資料量很大，因此 UDP 類型的傳輸需求變得更重要。

## Port number：分辨同一台主機上的不同程式

`IP address` 只能找到哪一台主機，但一台主機上可能同時有很多程式在通訊。

所以還需要 `Port number`。

可以這樣記：

- `IP address`：找到哪一台電腦。
- `Port number`：找到那台電腦上的哪一支程式或服務。

### Port 是邏輯連線，不是實體插孔

課堂影片中也提到：

- Port 不是實體線孔。
- Port 是程式與服務交換資料時使用的邏輯編號。
- 收到封包後，系統會根據 port 把資料交給對應的應用程式。

### Port number 範圍與分類

Port number 是 **16 bits**，所以範圍是：

- `0 ~ 65535`

常見分類：

| 類型 | 範圍 | 說明 |
|---|---:|---|
| Well-known ports | `0 ~ 1023` | 公認服務常用，不建議任意占用 |
| Registered ports | `1024 ~ 49151` | 可由特定應用或廠商註冊使用 |
| Dynamic / Private ports | `49152 ~ 65535` | 使用者端或暫時連線常用 |

### 常見 Port

| Port | 用途 |
|---:|---|
| `21` | FTP |
| `23` | Telnet |
| `25` | Email / SMTP |
| `53` | DNS |
| `80` | HTTP |
| `443` | HTTPS |

### 伺服器端與使用者端 Port

- 伺服器端常使用固定的 well-known port，例如網站常用 `80` 或 `443`。
- 使用者端通常會由系統臨時分配一個 port。
- 一組連線可以用：
  - 來源 IP
  - 來源 Port
  - 目的 IP
  - 目的 Port
  來區分。

## `netstat`：觀察自己電腦的連線

老師課堂中示範用 `netstat` 觀察電腦目前有哪些網路連線。

可以看到：

- 本機 IP 與本機 Port。
- 遠端 IP 或網域與遠端 Port。
- 連線狀態。
- 同一台電腦可能同時連到多個伺服器。
- 同一個網站或服務也可能開多條連線。

### 觀察重點

- 遠端 `:443` 通常代表 HTTPS 連線。
- 遠端 `:80` 通常代表 HTTP 連線。
- `127.0.0.1` 是本機 loopback，代表電腦自己跟自己通訊。
- 透過 port 可以看出資料大概是交給哪類服務處理。

## UDP Header

老師後半段開始講 UDP 的封包格式。

UDP Header 很簡單，固定 **8 bytes**，主要有 4 個欄位，每個欄位 16 bits：

| 欄位 | 長度 | 用途 |
|---|---:|---|
| Source Port | 16 bits | 來源 port |
| Destination Port | 16 bits | 目的 port |
| Length | 16 bits | UDP header + data 的總長度 |
| Checksum | 16 bits | 檢查資料是否可能出錯 |

### UDP Checksum

- UDP 雖然不保證送達，也不負責重傳，但仍有 `Checksum`。
- `Checksum` 用來檢查資料在傳輸過程中是否可能發生錯誤。
- 如果 checksum 不符合，代表資料可能損壞。
- 但 UDP 的重點不是「確保一定正確送達」，所以它不像 TCP 那樣提供完整可靠傳輸機制。

### Wireshark 觀察

老師示範封包擷取時，可以打開 UDP 封包看：

- Source Port
- Destination Port
- Length
- Checksum
- 封包中的資料內容

這可以幫助理解：課本上的 header 欄位不是抽象名詞，而是真的會出現在封包裡。

## TCP Header 與可靠傳輸概念

TCP Header 比 UDP 複雜很多，因為 TCP 要提供更完整的控制功能。

這次錄音主要先講兩個 TCP 重點：

- `Sequence number`
- `Acknowledgement number`（ACK number）

老師也預告後面還會再講：

- Sliding window
- 更完整的 TCP 控制機制

## Sequence number

`Sequence number` 用來標記資料在整個資料流中的位置。

TCP 不是只看「第幾個封包」，而是以 **byte** 為單位來編號。

用途：

- 確認資料順序。
- 判斷是否有資料遺失。
- 收到重複資料時可以辨識。
- 幫助接收端把資料重新組回正確順序。

### 為什麼要有序號

網路傳輸時可能發生：

- 封包遺失。
- 封包晚到。
- 封包重複。
- ACK 回應遺失。

有序號後，接收端就能知道：

- 這是不是我預期收到的下一段資料？
- 這段資料是不是已經收過？
- 中間是不是少了一段？

## ACK number

`ACK number` 用來告訴對方：「我已經收到哪裡，下一個希望收到哪個 byte。」

可以記成：

> ACK number = 下一個期望收到的序號

如果 A 傳送從序號 `60000` 開始、長度 `1000 bytes` 的資料，B 收到後會回：

- `ACK = 61000`

意思是：

- 我已經收到 `60000 ~ 60999`。
- 下一個請從 `61000` 開始送。

### 雙向傳輸時要分開看

TCP 連線中，雙方都可以傳資料。

所以：

- A 傳給 B 的資料，有 A 自己的 sequence number。
- B 傳給 A 的資料，也有 B 自己的 sequence number。
- ACK number 是針對「對方傳來的資料」做確認。
- Sequence number 與 ACK number 會互相搭配，但不是同一個數字。

## 重傳機制

老師用遺失情境說明 TCP 為什麼可靠。

可能情況：

1. 發送端送出的資料封包遺失。
2. 接收端收到資料，但回傳的 ACK 遺失。
3. 發送端等太久沒收到 ACK。
4. 發送端判斷可能出問題，就重新傳送。

重傳時，因為 sequence number 相同，接收端可以知道：

- 這是同一份資料。
- 如果之前已經收過，就不要重複組進資料流。
- 如果之前沒收過，就補上遺失的那一段。

## 緩衝區與視窗概念

錄音中也提到網路卡或接收端會有緩衝區，用來暫放收到的資料。

TCP 需要知道：

- 對方還能收多少資料？
- 目前可不可以繼續送？
- 送太快會不會讓對方來不及處理？

這會連到後續的 `sliding window`（滑動視窗）概念：

- 視窗大小代表目前可以連續傳送多少資料。
- 接收端用視窗資訊告訴傳送端自己的接收能力。
- 這可以避免傳送端一次塞太多資料造成壅塞或遺失。

## 怎麼記

- `IP`：找到哪一台主機。
- `Port`：找到主機上的哪一支程式。
- `TCP`：可靠、確認、重傳、順序、成本高。
- `UDP`：快速、簡單、即時、不保證送達。
- `SYN / ACK`：TCP 建立連線與確認資料的重要訊號。
- `Sequence number`：資料從哪個 byte 開始。
- `ACK number`：下一個期望收到的 byte。
- `UDP Header`：4 欄位、固定 8 bytes。

## 課堂例子整理

### 看網頁或下載檔案

- 網頁內容、圖片、檔案資料通常不能亂掉。
- 遺失一部分可能導致檔案不能用或頁面顯示錯誤。
- 適合使用 `TCP`。

### 看影片或聽語音

- 即時性很重要。
- 少量封包遺失可能只是短暫破音或畫面不完整。
- 如果一直重傳，反而會卡住。
- 適合使用 `UDP` 或偏 UDP 思維的傳輸方式。

### 同一台電腦多個程式同時上網

- 只靠 IP 只能知道資料到哪台電腦。
- 需要 port 才知道資料要交給瀏覽器、FTP、Email，或其他程式。

### 用 `netstat` 看連線

- 可以看到本機與遠端的 IP:Port。
- `443` 常代表 HTTPS。
- `80` 常代表 HTTP。
- `127.0.0.1` 代表本機 loopback。
