# TCP 與 UDP

## 傳輸層在做什麼

第八課的 `IP address` 解決的是「資料要送到哪一台主機」。

第九課的傳輸層進一步處理：

- 這份資料要交給那台主機上的哪一個應用程式？
- 資料有沒有遺失？
- 資料順序有沒有錯？
- 如果沒收到，要不要重傳？
- 這個應用比較需要可靠性，還是比較需要即時性？

所以傳輸層提供了兩個最常見的協定：`TCP` 與 `UDP`。

## TCP 與 UDP 快速比較

| 項目        | TCP                           | UDP                    |
| --------- | ----------------------------- | ---------------------- |
| 全名        | Transmission Control Protocol | User Datagram Protocol |
| 連線方式      | Connection-oriented           | Connectionless         |
| 傳送前是否建立連線 | 會，透過三向交握                      | 不會，直接送出                |
| 是否確認送達    | 會用 ACK 確認                     | 不保證                    |
| 是否維持順序    | 會用 sequence number            | UDP 本身不保證              |
| 遺失時是否重傳   | 會                             | UDP 本身不重傳              |
| Header 成本 | 較高，最少 20 bytes                | 很低，固定 8 bytes          |
| 特色        | 可靠、順序、控制多                     | 快速、簡單、延遲低              |
| 常見用途      | 網頁、檔案下載、Email、SSH             | DNS、語音、影音、遊戲、即時資料      |

## 怎麼選

- 資料**不能缺、不能亂、不能錯**：通常選 `TCP`。
- 資料更在意**即時性**，少量遺失可以接受：通常選 `UDP`。

例如：

- 下載軟體或傳檔案時，少一段資料可能讓檔案壞掉，所以適合 TCP。
- 語音或直播時，偶爾少一點封包可能只是短暫破音或畫面不完整；如果每段都等待重傳，反而會卡住，所以常偏向 UDP。

## TCP 的核心想法

TCP 會把應用程式交下來的資料看成連續的 **byte stream**。

它會做：

- 建立連線。
- 拆成多個 TCP segment。
- 為資料編 sequence number。
- 用 ACK number 確認已收到的位置。
- 發生遺失時重傳。
- 用 window 控制傳送速度。
- 必要時做 flow control 與 congestion control。

TCP 的成本比較高，但換來的是可靠與順序。

## UDP 的核心想法

UDP 比較像 **fire and forget**。

它會做：

- 把資料加上簡單 header。
- 標明來源 port、目的 port、長度與 checksum。
- 交給網路層送出去。

它不負責：

- 建立連線。
- 確認對方收到。
- 重傳遺失資料。
- 幫你重排順序。
- 幫你做流量控制。

如果應用程式真的需要這些功能，就要在應用層自己補上。

## 一句話記法

- `TCP`：先確認、再傳送；慢一點，但可靠。
- `UDP`：直接送；快一點，但不保證。
