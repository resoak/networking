# 第十課 ARP、RARP、ICMP 習題答案

> 範圍：10.3 問題與討論  
> 說明：以下答案以常見的 Ethernet + IPv4 環境為主。

---

## 一、填充題

1. **ARP**；**RARP**  
   ARP 用來將 IP 位址解析成實體位址，RARP 則是由實體位址反查 IP 位址。

2. **乙太網路標頭的目的位址欄位**  
   ARP Request 會以 `FF-FF-FF-FF-FF-FF` 作為乙太網路目的 MAC 位址，表示廣播給同一區域網路內所有主機。

3. **錯誤訊息** 與 **查詢訊息**

4. **類型 Type**、**代碼 Code** 與 **檢查碼 Checksum**

5. **回聲**

6. **目的端無法抵達（Destination Unreachable）**

7. **來源端抑制（Source Quench）**

8. **傳輸逾時（Time Exceeded）**

---

## 二、選擇題

1. **A. ARP**  
   ARP 是將 IP 位址映射為實體位址的協定。

2. **B. IP**  
   TCP/IP 架構中的邏輯位址是 IP 位址。

3. **B. IP 與 MAC 的對照表**  
   ARP Table 儲存 IP 位址與 MAC 位址的對應關係。

4. **C. 只知目的端的 IP 位址**  
   只知道目的端 IP 位址時，通常是使用 ARP 查 MAC；RARP 是已知自己的 MAC、反查自己的 IP。

5. **A. Source Port Number**  
   Source Port Number 是傳輸層 TCP/UDP 的欄位，不是 ARP/RARP 標頭欄位。

6. **C. ICMP**  
   ICMP 用來補強 IP 未提供的錯誤回報、控制與查詢功能。

7. **B. 來源端主機傳送封包的資料太快**  
   Source Quench 用於通知來源端降低傳送速率。

8. **D. Redirect**  
   Redirect 用於通知主機改用較佳的路由或閘道。

9. **B. 1200**  
   計算如下：
   - 發送端原始時戳 `T1 = 101800`
   - 接收端接收時戳 `T2 = 102400`
   - 接收端傳送時戳 `T3 = 103900`
   - 發送端接收時戳 `T4 = 105700`
   - 來回總時間：`T4 - T1 = 105700 - 101800 = 3900`
   - 接收端處理時間：`T3 - T2 = 103900 - 102400 = 1500`
   - 實際網路往返延遲：`3900 - 1500 = 2400`
   - 單趟延遲：`2400 / 2 = 1200`

## 三、問答題

### 1. ARP、RARP、MAC、ICMP 的英文全名

1. **ARP**：Address Resolution Protocol，位址解析協定。
2. **RARP**：Reverse Address Resolution Protocol，反向位址解析協定。
3. **MAC**：Media Access Control，媒體存取控制。
4. **ICMP**：Internet Control Message Protocol，網際網路控制訊息協定。

### 2. 邏輯位址與實體位址的意義與差異

**邏輯位址**通常指 IP 位址，由網路層使用，可依網路環境或設定而改變，用來表示主機在網路中的位置。  
**實體位址**通常指 MAC 位址，由資料鏈結層使用，多半燒錄或設定在網路介面卡上，用來在同一區域網路內實際傳送訊框。  
兩者主要差異在於：IP 位址負責跨網路的邏輯定位，MAC 位址負責區域網路內的實際硬體傳送。

### 3. Ethernet 標頭 Etype 欄位對 ARP 與 RARP 的設定值

- **ARP**：`0x0806`
- **RARP**：`0x8035`

Etype 欄位用來指出乙太網路訊框所承載的上層協定類型。

### 4. 為何需要 ARP、何謂 ARP Table、ARP 如何運作

IP 封包在區域網路中仍必須封裝成資料鏈結層訊框才能傳送，而訊框需要目的端 MAC 位址；因此當主機只知道目的 IP 位址時，就需要 ARP 找出對應的 MAC 位址。

**ARP Table** 是主機或路由器中的快取表，用來記錄 IP 位址與 MAC 位址的對應關係。

ARP 的基本運作流程：

1. 主機先查詢自己的 ARP Table。
2. 若已有目的 IP 對應的 MAC 位址，就直接使用該 MAC 傳送訊框。
3. 若沒有對應資料，主機會送出 ARP Request 廣播。
4. 擁有該 IP 位址的主機收到後，以 ARP Reply 回覆自己的 MAC 位址。
5. 發送端將此對應關係存入 ARP Table，再進行資料傳送。

### 5. 為何需要 RARP、與 ARP 的差異、應用場合、RARP 伺服器與運作方式

RARP 用於主機知道自己的 MAC 位址，但不知道自己的 IP 位址時，向網路上的伺服器查詢可使用的 IP 位址。

ARP 與 RARP 的差異：

- **ARP**：已知 IP，查詢 MAC。
- **RARP**：已知 MAC，查詢 IP。

RARP 早期常用於**無磁碟工作站**或剛啟動、尚未設定 IP 的主機。  
**RARP 伺服器**會保存 MAC 位址與 IP 位址的對照表，並負責回覆用戶端的 RARP 查詢。

RARP 的基本運作流程：

1. 用戶端以廣播方式送出 RARP Request，附上自己的 MAC 位址。
2. RARP 伺服器查詢對照表。
3. 若找到對應 IP，就以 RARP Reply 回覆該主機可使用的 IP 位址。
4. 用戶端取得 IP 後，便可進行 IP 通訊。

目前實務上 RARP 多已被 BOOTP 或 DHCP 取代。

### 6. ARP/RARP 標頭欄位大小、功能與區分方式

在 Ethernet + IPv4 中，ARP/RARP 常見標頭格式如下：

| 欄位 | 大小 | 功能 |
|---|---:|---|
| Hardware Type | 16 bits | 硬體類型，例如 Ethernet 為 `1` |
| Protocol Type | 16 bits | 上層協定類型，例如 IPv4 為 `0x0800` |
| Hardware Address Length | 8 bits | 實體位址長度，Ethernet MAC 為 `6` bytes |
| Protocol Address Length | 8 bits | 協定位址長度，IPv4 為 `4` bytes |
| Operation / Opcode | 16 bits | 操作碼，用來表示請求或回覆類型 |
| Sender Hardware Address | 依 HLEN 而定 | 發送端實體位址 |
| Sender Protocol Address | 依 PLEN 而定 | 發送端協定位址，例如 IP |
| Target Hardware Address | 依 HLEN 而定 | 目的端實體位址 |
| Target Protocol Address | 依 PLEN 而定 | 目的端協定位址，例如 IP |

ARP 與 RARP 主要可由 **Operation / Opcode** 區分：

- `1`：ARP Request
- `2`：ARP Reply
- `3`：RARP Request
- `4`：RARP Reply

若以乙太網路標頭來看，也可由 Etype 區分：ARP 為 `0x0806`，RARP 為 `0x8035`。

### 7. ARP 封包訊息解析

以 Ethernet + IPv4 的 ARP Request 為例，可解析如下：

1. **乙太網路目的位址**：`FF-FF-FF-FF-FF-FF`，代表廣播。
2. **乙太網路來源位址**：發送端主機的 MAC 位址。
3. **Etype**：ARP 為 `0x0806`。
4. **Hardware Type**：Ethernet 通常為 `1`。
5. **Protocol Type**：IPv4 為 `0x0800`。
6. **Hardware Address Length**：MAC 長度為 `6` bytes。
7. **Protocol Address Length**：IPv4 長度為 `4` bytes。
8. **Operation**：ARP Request 為 `1`，ARP Reply 為 `2`。
9. **Sender Hardware Address**：發送端 MAC。
10. **Sender Protocol Address**：發送端 IP。
11. **Target Hardware Address**：ARP Request 時通常為 `00-00-00-00-00-00`，表示尚未知曉。
12. **Target Protocol Address**：欲查詢的目標 IP。

若為 ARP Reply，則會改為單播回覆，並在 Target Hardware Address 與相關欄位中填入實際位址資訊。

### 8. IP 協定與 ARP/RARP 之間的關係

IP 協定使用邏輯位址進行封包遞送，但實際在區域網路中傳送資料時，需要資料鏈結層的實體位址。ARP/RARP 就是協助 IP 與實體網路之間進行位址轉換的機制。

- **ARP**：讓 IP 封包能找到下一站或目的端的 MAC 位址。
- **RARP**：讓只知道自己 MAC 位址的主機取得 IP 位址。

因此 ARP/RARP 可視為 IP 與資料鏈結層之間的重要輔助協定。

### 9. ARP Request 中 Ethernet 目的位址與 ARP 目的端硬體位址的差異

`FF-FF-FF-FF-FF-FF` 是**乙太網路標頭的目的 MAC 位址**，代表這個訊框要廣播給同一區域網路內所有主機。

`00-00-00-00-00-00` 是 **ARP 標頭中的目的端硬體位址欄位**，代表發送端目前還不知道目標主機的 MAC 位址，所以先以 0 填入。

簡單來說：

- `FF-FF-FF-FF-FF-FF` 是用來「送給所有人看」。
- `00-00-00-00-00-00` 是用來表示「目標 MAC 尚未知」。

### 10. ICMP 與 IP 的關係、主要目的與常見功能

ICMP 是 TCP/IP 協定族的一部分，通常封裝在 IP 封包中傳送。IP 本身只負責盡力傳送，不保證可靠送達，也不主動提供完整錯誤回報；ICMP 則用來補足錯誤通知、控制訊息與診斷功能。

常用指令確實會使用 ICMP，例如：

- **ping**：使用 Echo Request / Echo Reply 測試目的端是否可達。
- **traceroute / tracert**：利用 TTL 遞減後產生的 Time Exceeded 訊息追蹤路徑。

常見 ICMP 功能包括：

- **Echo Request / Echo Reply**：測試連線可達性。
- **Destination Unreachable**：通知目的端、網路、通訊埠等無法抵達。
- **Source Quench**：通知來源端降低傳送速率；此機制已較少或不再使用。
- **Redirect**：通知主機改用較佳路由。
- **Time Exceeded**：封包存活時間過期或分段重組逾時。
- **Parameter Problem**：IP 標頭欄位有錯誤。
- **Timestamp Request / Reply**：用於時間戳與延遲估算。
- **Address Mask Request / Reply**：查詢網路遮罩，現多由 DHCP 取代。

### 11. ICMP Type、Code 與標頭格式

**Type** 欄位用來表示 ICMP 訊息的大類型，例如 Echo、Destination Unreachable、Redirect、Time Exceeded 等。依功能可大致分為：

1. **錯誤回報類訊息**：如 Destination Unreachable、Time Exceeded、Parameter Problem。
2. **查詢／診斷類訊息**：如 Echo Request / Reply、Timestamp Request / Reply。

**Code** 欄位則用來表示同一 Type 底下更細部的原因或狀態。例如 Destination Unreachable 這個 Type 內，Code 可再表示網路不可達、主機不可達、通訊埠不可達等不同原因。

ICMP 基本標頭格式：

| 欄位 | 大小 | 功能 |
|---|---:|---|
| Type | 8 bits | ICMP 訊息類型 |
| Code | 8 bits | 類型下的細部分類或原因 |
| Checksum | 16 bits | ICMP 訊息檢查碼 |
| 其他欄位 | 依 Type 不同 | 例如 Identifier、Sequence Number、Pointer、Gateway Address 等 |
| Data | 依 Type 不同 | 錯誤訊息常包含原 IP 標頭與前 64 bits 資料 |

### 12. ICMP 錯誤訊息中包含原 IP 標頭與 64 位元資料的用途

ICMP 錯誤訊息會附上造成錯誤的原始 IP 標頭，以及原資料的前 64 bits，目的是讓來源端判斷是哪一個封包造成問題。

原 IP 標頭可提供來源 IP、目的 IP、協定類型等資訊；前 64 bits 資料通常會包含 TCP 或 UDP 標頭的一部分，例如來源埠與目的埠，因此作業系統可以把錯誤對應回正確的應用程式或連線。

### 13. Source Quench 訊息的運作與格式

Source Quench 用於路由器或目的端因壅塞、緩衝區不足，或來源端傳送過快而無法順利處理封包時，通知來源端降低傳送速率。

其 ICMP 格式重點如下：

| 欄位 | 內容 |
|---|---|
| Type | `4` |
| Code | `0` |
| Checksum | ICMP 檢查碼 |
| Unused | 32 bits，通常填 0 |
| Data | 原 IP 標頭與前 64 bits 資料 |

來源端收到後應減慢傳送速度，避免加重網路壅塞。不過 Source Quench 在現代網路中已被視為過時機制。

### 14. Redirect 訊息的運作機制與格式

Redirect 用於路由器發現來源主機有更好的下一站路由時，通知該主機改用較佳的閘道。常見情況是主機把封包送給某路由器，但該路由器知道同一網路中另一台路由器更適合作為下一站。

其 ICMP 格式重點如下：

| 欄位 | 內容 |
|---|---|
| Type | `5` |
| Code | `0` 網路重新導向、`1` 主機重新導向、`2` TOS 與網路重新導向、`3` TOS 與主機重新導向 |
| Checksum | ICMP 檢查碼 |
| Gateway Internet Address | 建議改用的閘道 IP 位址 |
| Data | 原 IP 標頭與前 64 bits 資料 |

來源主機收到後，可更新自己的路由快取或路由選擇。

### 15. Time Exceeded 訊息的運作機制與格式

Time Exceeded 用於封包在網路中停留過久而被丟棄時，通知來源端。

主要情況有兩種：

1. 路由器每轉送一次封包就將 TTL 減 1，當 TTL 變成 0 時，路由器丟棄封包並送出 Time Exceeded。
2. 目的端在指定時間內無法完成 IP 分段重組，也會送出 Time Exceeded。

其 ICMP 格式重點如下：

| 欄位 | 內容 |
|---|---|
| Type | `11` |
| Code | `0` TTL 過期、`1` 分段重組逾時 |
| Checksum | ICMP 檢查碼 |
| Unused | 32 bits，通常填 0 |
| Data | 原 IP 標頭與前 64 bits 資料 |

traceroute / tracert 便是利用 TTL 遞增與 Time Exceeded 回應來推測路徑上的路由器。

### 16. Parameter Problem 訊息的運作機制與格式

Parameter Problem 用於路由器或目的端發現 IP 標頭中有錯誤欄位，導致封包無法正確處理時，通知來源端。

其 ICMP 格式重點如下：

| 欄位 | 內容 |
|---|---|
| Type | `12` |
| Code | 常見為 `0`，表示 Pointer 指出錯誤位置 |
| Checksum | ICMP 檢查碼 |
| Pointer | 指出原 IP 標頭中發生問題的位元組位置 |
| Unused | 24 bits，通常填 0 |
| Data | 原 IP 標頭與前 64 bits 資料 |

Pointer 欄位可協助來源端知道是哪一個 IP 標頭欄位發生問題。

### 17. Timestamp Request / Reply 的運作、格式與延遲計算

Timestamp Request / Reply 用於取得雙方時間戳，並估算傳輸延遲或時間差。

其 ICMP 格式重點如下：

| 欄位 | 內容 |
|---|---|
| Type | Request 為 `13`，Reply 為 `14` |
| Code | `0` |
| Checksum | ICMP 檢查碼 |
| Identifier | 識別碼，用來配對請求與回覆 |
| Sequence Number | 序列碼，用來區分不同封包 |
| Originate Timestamp | 發送端送出請求的時間 `T1` |
| Receive Timestamp | 接收端收到請求的時間 `T2` |
| Transmit Timestamp | 接收端送出回覆的時間 `T3` |

若發送端收到回覆時記錄時間為 `T4`，則可估算：

- 來回總時間：`T4 - T1`
- 接收端處理時間：`T3 - T2`
- 實際網路往返延遲：`(T4 - T1) - (T3 - T2)`
- 若假設來回路徑延遲相近，單趟延遲約為：

```text
[(T4 - T1) - (T3 - T2)] / 2
```

### 18. Information Request / Reply 的用途與不再使用的原因

Information Request / Reply 早期主要用來讓主機取得網路相關資訊，例如在尚未完整設定網路參數時，向網路設備查詢自己所在網路的資訊。

其 ICMP 類型為：

- **Information Request**：Type `15`
- **Information Reply**：Type `16`

目前已不再常用，主要原因是後來有更完整、更安全且功能更強的設定機制，例如 BOOTP 與 DHCP；此外，早期依賴 classful network 的設計也已不符合現代網路環境。

### 19. Address Mask Request / Reply 的功能與格式

Address Mask Request / Reply 用於主機查詢所在網路應使用的子網路遮罩。主機可送出 Address Mask Request，由路由器或伺服器回覆正確的 Address Mask。

其 ICMP 格式重點如下：

| 欄位 | 內容 |
|---|---|
| Type | Request 為 `17`，Reply 為 `18` |
| Code | `0` |
| Checksum | ICMP 檢查碼 |
| Identifier | 識別碼，用來配對請求與回覆 |
| Sequence Number | 序列碼，用來區分不同封包 |
| Address Mask | 32 bits 的子網路遮罩，例如 `255.255.255.0` |

Address Mask Request 通常可用廣播送出；Reply 則回覆該網路的遮罩值。不過在現代網路中，子網路遮罩多由 DHCP 一併提供，因此此類 ICMP 訊息也已較少使用。
