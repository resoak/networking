## 導覽

- 這一課的起點是：`IP address` 只能告訴我們資料要去的邏輯位置。
- 但在同一個區域網路內，實際送出乙太網路 frame 時，第二層還需要 `MAC address`。
- 所以主機在送資料前，要先解決兩件事：
  - 我要找誰的 `MAC address`？
  - 現在網路到底通不通，問題卡在哪裡？
- 這就連到兩組重點：
  - `ARP / RARP`：處理位址解析。
  - `ICMP`：處理錯誤回報、偵測與診斷。

## 課堂主軸

老師先從區域網路的實際傳送開始講。主機雖然知道對方的 `IP address`，但如果要在區網中送出 frame，仍然要先知道對方或下一跳設備的 `MAC address`。因此 `ARP` 的工作，就是把第三層的邏輯位址轉成第二層會用到的實體位址。

接著老師說明 `ARP` 為什麼常和 `cache` 一起出現。主機不會每送一次資料就重新廣播一次，它會先查 `ARP cache`。如果快取裡已有 `IP ↔ MAC` 對照，就能直接封裝資料；如果沒有，才送出 `ARP request` 廣播詢問。

這裡也補上同網段與不同網段的差別：

- **同網段**：直接查目標主機的 `MAC address`。
- **不同網段**：不會直接找遠端主機的 `MAC address`，而是先找 `default gateway` 或 Router 的 `MAC address`。

老師接著補充 `RARP` 的概念。`RARP` 是反方向的解析，也就是已知 `MAC address`，想反查 `IP address`。這是較早期的做法，現在通常只要知道它的方向與歷史用途即可。

後半段課程轉到 `ICMP`。老師強調 `IP` 本身偏向盡力傳送，但網路中還是需要一套機制告訴我們「主機是否可達」、「路徑是否異常」、「封包是不是被丟掉了」。這就是 `ICMP` 的工作。它不是用來搬運一般應用資料，而是用來回報狀態、錯誤與測試結果。

## ARP 運作流程整理

老師把 ARP 流程講得很白話，可以整理成下面幾步：

1. 主機先查自己的 `ARP cache`。
2. 如果查得到目標 IP 對應的 MAC，就直接送資料。
3. 如果查不到，就送出 `ARP request` 廣播。
4. 區網中所有主機都會看到這個廣播。
5. 只有目標 IP 對應的那台主機會回 `ARP reply`。
6. 發問主機把新對照寫進 `ARP cache`。
7. 之後再傳資料時，就能直接用正確的 MAC。

老師也搭配 Wireshark 觀察，讓大家看到：

- `ARP request` 會先出現在封包清單中。
- 回應端接著送出 `ARP reply`。
- 查表成功後，再開始看到後續的 `ping` 或其他資料封包。

## ARP Cache 與指令重點

錄音 14 與 16 都有示範指令。核心觀念是：

- `arp -a`：查看目前快取內容。
- `arp -d`：刪除快取項目。
- `arp -s`：手動加入靜態項目。

課堂也提到 `dynamic` 與 `static`：

- `dynamic` 是系統透過 ARP 自動學到的，會過期。
- `static` 是手動設定的，不會像動態項目那樣自動更新。

老師用兩種情境幫大家分辨：

- `ping` 同網段主機後，再看 `arp -a`，會看到那台主機的對照。
- `ping 8.8.8.8` 這類不同網段位址後，再看 `arp -a`，看到的通常不是 `8.8.8.8` 的 MAC，而是 `default gateway` 的 MAC。

## ICMP 的角色

老師把 `ICMP` 定位成「網路控制與診斷的語言」。當路上有錯誤，或我們想主動測試連線時，常會靠 `ICMP` 來完成。

`ICMP` 的重要用途包括：

- 回報封包錯誤。
- 告訴主機目標是否可達。
- 協助測試對方是否在線上。
- 協助觀察封包路徑。

課堂中特別提到 `ping` 與 `traceroute`：

- `ping`：確認主機是否可達，並量測基本往返情況。
- `traceroute`：看封包一路經過哪些中間設備。

## ICMP Header 與封包觀察

老師把 `ICMP header` 的重點濃縮成幾個欄位：

- `Type`：這是哪一種 ICMP 訊息。
- `Code`：在同一個 Type 下，再細分原因。
- `Checksum`：檢查封包內容是否可能出錯。
- `Identifier`：同一次 `ping` 常共用同一組識別值。
- `Sequence Number`：同一次 `ping` 的每一對 request / reply 會逐步加一。

Wireshark 觀察時，老師提醒可以同時看三層：

- 第二層：來源 MAC、目的 MAC。
- 第三層：來源 IP、目的 IP。
- ICMP：`Type`、`Code`、`identifier`、`sequence number`。

## 常見 ICMP 訊息整理

課堂主要提到：

- `Type 8`：`Echo Request`，也就是 `ping` 發出去的請求。
- `Type 0`：`Echo Reply`，也就是對方回來的回應。
- `Type 3`：`Destination Unreachable`，代表目標不可達，但還要看 `Code` 才知道是網路、主機、還是協定層問題。
- `Type 5`：`Redirect`，告訴主機有更合適的下一跳。
- `Type 11`：`Time Exceeded`，常和 `traceroute` 有關。
- `Type 12`：`Parameter Problem`，表示封包欄位或格式可能有問題。

另外老師也帶過一些較早期或概念性訊息，例如 `Source Quench`、`Timestamp`、`Address Mask`，考試或複習時知道大意即可。

## 網路診斷流程

錄音 16 最實用的部分，是老師示範「不能上網時要怎麼一步一步查」。順序如下：

1. `ping 127.0.0.1`，先測本機 `TCP/IP` 軟體堆疊。
2. `ping` 自己網卡的 IP，確認本機網卡是否正常。
3. `ping` Router 或無線 AP 的 IP，確認本機到閘道是否通。
4. `ping 8.8.8.8` 這類外部 IP，確認對外路由是否正常。
5. `ping www.google.com` 這類網域名稱，確認 DNS 解析是否正常。

這個流程的好處是把問題逐步縮小。只要卡在哪一步，就能大致推測問題更可能出在本機、網卡、Router、ISP，還是 DNS。

## 怎麼記

- `ARP`：`IP → MAC`。
- `RARP`：`MAC → IP`，較舊、較少見。
- `ARP cache`：把查過的對照先記起來。
- 同網段找目標主機 MAC，不同網段找 `default gateway` MAC。
- `ICMP`：錯誤回報、偵測、控制訊息。
- `ping` 看可達性。
- `traceroute` 看經過哪些路由器。
- 診斷順序：`127.0.0.1 → 本機 IP → Router → 外部 IP → 網域名稱`。
